# Dependency Updates

DataKnobs uses an automated weekly dependency-update workflow that does
more than refresh `uv.lock` -- it also surfaces CVE findings at *both*
the upgraded resolve and the floor (lowest-direct) resolve, and runs
the full quality suite against both. The output is one PR per week
with everything reviewers need to triage.

## Automated Update Workflow

The workflow lives at `.github/workflows/dependency-update.yml`.

### Schedule and Trigger

- **Runs weekly** on Monday at 15:00 UTC (9 AM MDT / 8 AM MST)
- Can also be **triggered manually** via `workflow_dispatch` from the
  GitHub Actions UI
- A 2-hour debounce step skips the run if the `update-dependencies`
  branch was pushed within the previous 2 hours (avoids duplicate
  work when manually retried)

### What the Workflow Does

The workflow audits **two distinct resolves** of the workspace:

| Resolve | What it represents | Generated by |
|---|---|---|
| **Upgraded** (`uv.lock`) | Latest versions permitted by every direct + transitive constraint -- what `uv lock --upgrade` produces | `uv lock --upgrade` |
| **Floor** (`uv-lowest.lock`) | Every direct dep pinned to its declared lower bound -- what a fresh consumer install can land on without inheriting our `uv.lock` | `uv lock --resolution lowest-direct` |

Each resolve is audited for CVEs and validated for quality:

1. **Refresh** `uv.lock` via `uv lock --upgrade`, then snapshot it as
   `uv.lock.initial` -- the bit-identical file that will be committed
   to the PR after both audits and validations finish
2. **Sync** the upgraded resolve via `uv sync --all-packages`
3. **Audit upgraded resolve** with `osv-scanner` against `uv.lock`
4. **Validate upgraded resolve** by running `bin/dk pr --all`
5. **Generate floor resolve** by running
   `uv lock --resolution lowest-direct`, copying the result to
   `uv-lowest.lock`, and restoring `uv.lock` from `uv.lock.initial`
6. **Audit floor resolve** with `osv-scanner` against `uv-lowest.lock`
7. **Sync floor resolve** by overwriting `uv.lock` with
   `uv-lowest.lock` and running `uv sync --all-packages --frozen`
   (frozen sync installs exactly the lockfile we just audited, with
   no re-resolution)
8. **Validate floor resolve** by running `bin/dk pr --all`
9. **Restore upgraded resolve** by moving `uv.lock.initial` back to
   `uv.lock` and frozen-syncing, so the PR commits the exact lockfile
   that was audited at the start of the workflow
10. **Open the PR** with all four results -- two audits + two
    validations -- in the PR body

### Why Two Resolves?

The upgraded-resolve audit catches CVEs that affect the versions in
`uv.lock`. But many consumers install dataknobs without inheriting
our lockfile -- they get whatever fresh resolution their resolver
picks, bounded only by our `pyproject.toml` `>=` floors. If a floor
permits a vulnerable version, those consumers can land on it.

The floor-resolve audit answers: **"what could a fresh consumer
install actually pull in if their resolver picked our minimum
allowed versions?"** Findings here mean a `pyproject.toml` floor
should be bumped.

## Reviewing the Weekly PR

The weekly PR body contains four results from the four steps above.
Treat each as its own decision.

### Upgraded-Resolve Vulnerabilities

CVEs flagged here apply to the versions currently resolved in
`uv.lock`. If any are found, the upgrade itself isn't safe to merge
without a fix. Either:

- The flagged dep needs a constraint update (rare -- usually means
  the latest version still has a CVE), **or**
- The CVE is in a dep that's optional and unused in our test path,
  in which case it can ship merged with a follow-up triage item

### Upgraded-Resolve Validation

This is `bin/dk pr --all` against the upgraded resolve. If it fails,
some new dep version broke our code. The PR body shows the last
~100 lines of the failure log; check the workflow logs for the full
output. Common patterns:

- A library deprecated an API we still call -- fix our code to use
  the new API, or pin the dep below the deprecation version with a
  comment explaining why
- A library tightened type signatures and we now fail mypy -- update
  our annotations
- A test fixture broke because of a behavior change in an underlying
  library -- update the test

### Floor-Resolve Vulnerabilities

CVEs flagged here apply to versions a fresh consumer install could
pull in. **The fix is to bump the affected `pyproject.toml` floor**
with an inline CVE-rationale comment (see "Bumping a Floor" below).

If a finding is *deferrable* -- e.g., the fix version is a release
candidate, the affected dep is dev-only with no production impact,
or the fix would require a major-version bump too risky to bundle
with a security PR -- note it in the merge commit and leave it for
a follow-up. The workflow will keep flagging it weekly until the
floor is bumped.

### Floor-Resolve Validation

This is `bin/dk pr --all` against the floor resolve. Failures here
mean our code has drifted past what our minimum declared
dependencies actually support -- e.g., we use a method that didn't
exist in the floor version of a library. The fix is either:

- **Use a more conservative API** that works at the floor, **or**
- **Bump the floor** so the API we use is guaranteed available

Floor-validation failures are usually a signal that the floor on
the offending dep was set too permissive when the dep was first
added, and time has eroded compatibility. Bump it with a comment
referencing the API requirement, e.g., `pandas>=2.1.0 # uses
DataFrame.assign(... ignore_index=True) added in 2.1.0`.

## Bumping a Floor

When a floor-resolve audit (or any other CVE source) calls for a
floor bump:

### 1. Edit the `pyproject.toml` Floor With an Inline Comment

Match the precedent at `packages/llm/pyproject.toml:34`:

```toml
"torch>=2.6.0",  # CVE-2025-32434 (RCE in torch.load) fixed in 2.6.0
```

For multiple CVEs in one bump, list the OSV IDs (preferred) or CVE
IDs:

```toml
"nltk>=3.9.4",  # GHSA-rf74-v2fm-23pw, CVE-2026-33230, CVE-2026-33231 fixed in 3.9.4
```

For a bump where the floor is sweeping a transitive CVE via an
upstream constraint, name the transitive and the chain:

```toml
"httpx>=0.27.0",  # GHSA-vqfr-h8mv-ghfj in transitive h11<0.16 (CVSS 9.1) swept by httpcore 1.x (required by httpx>=0.27)
```

### 2. Add a Security Entry to the Package's CHANGELOG `## Unreleased`

Each affected package gets a `### Security` block explaining the
bump. The entry stays under `## Unreleased` until the next release;
when you run `bin/release-helper.sh bump`, it becomes the release
section automatically.

```markdown
## Unreleased

### Security
- Bumped minimum `nltk` requirement from `>=3.7` to `>=3.9.4` to
  exclude versions affected by GHSA-rf74-v2fm-23pw, CVE-2026-33230,
  and CVE-2026-33231 (one DoS, two in the WordNet browser HTTP
  component).
```

### 3. Regenerate `uv.lock`

```bash
uv lock
```

This propagates the new floor through the workspace lockfile.

### 4. Verify the Bump Closed the Finding

Re-run the floor audit locally:

```bash
bin/audit-floor.sh
```

The CVE you bumped for should no longer appear. If it does, the
floor isn't high enough -- check the OSV `FIXED VERSION` column and
bump higher.

### 5. Skip Version Bumps Until Release

**Do not bump package versions or run `bin/release-helper.sh bump`
just because a floor moved.** Floor bumps accumulate under
`## Unreleased` and ship together with whatever else is queued for
the next release. Bumping versions on every floor change creates
release churn and decouples security work from the release cadence
that maintainers actually drive.

When you're ready to release, `bin/release-helper.sh bump` walks
each package, shows what changed under `## Unreleased`, prompts
for the bump type (patch/minor/major), and synchronizes
`pyproject.toml`, `__init__.py __version__`, `.dataknobs/packages.json`,
and the docs version table.

## The `bin/audit-floor.sh` Helper

The same floor-level audit the workflow runs weekly is available as
a local script:

```bash
bin/audit-floor.sh
```

It:

1. Backs up `uv.lock` to `uv.lock.upgraded`
2. Regenerates `uv.lock` as the lowest-direct resolve
3. Copies the floor lockfile aside as `uv-lowest.lock`
4. Restores `uv.lock` to the upgraded resolve
5. Runs `osv-scanner scan --lockfile uv.lock:uv-lowest.lock` and
   prints the table
6. Cleans up `uv-lowest.lock` (it's gitignored anyway)

**Prerequisites:**

- `uv` on PATH (already required for dataknobs development)
- `osv-scanner` on PATH:
    - macOS: `brew install osv-scanner`
    - Linux: download the binary from the release page referenced in
      `.github/workflows/dependency-update.yml` (currently v2.3.3)

**Use it:**

- Before pushing a PR that bumps a floor, to confirm the bump closed
  the targeted CVE and didn't leave others
- Before opening a security-related PR, to see what other floors
  are also vulnerable -- you may want to bundle them
- Anytime you want a snapshot of the workspace's CVE exposure at the
  floor

The script's `trap cleanup EXIT` ensures `uv.lock` is always
restored even if `osv-scanner` fails.

## Reviewing a Standard (Non-CVE) Dependency Update

The original review flow still applies for the upgraded-resolve
diff in the weekly PR.

### 1. Check Out the PR Branch

```bash
git checkout main
git fetch
git pull
git branch -d update-dependencies
git checkout -b update-dependencies origin/update-dependencies
```

!!! tip
    The `git branch -d` step deletes any leftover local branch from
    a previous update cycle. Failure to delete a non-existent branch
    is harmless.

### 2. Sync the Updated Dependencies

```bash
uv sync --all-packages --all-groups
```

!!! warning "Stale .venv"
    If the repository's working directory has moved (e.g., you
    renamed or relocated the repo folder), the `.venv` may contain
    stale absolute paths. Delete and recreate:

    ```bash
    rm -rf .venv
    uv sync --all-packages --all-groups
    ```

### 3. Review the Dependency Changes

```bash
bin/dk deps
```

This compares the current `uv.lock` against `main`:

- **Updated** packages with old → new versions (major-version bumps
  flagged with `MAJOR`)
- **Added** packages (new transitive dependencies)
- **Removed** packages (dropped transitive dependencies)

Pay attention to:

- **Major version bumps** -- read the upstream changelog
- **New or removed packages** -- understand why transitive deps
  shifted (often because an upstream changed its own constraints)
- **Large minor-version jumps** -- may include behavioral changes

### 4. Re-Run Full Quality Checks Locally

The workflow already ran `bin/dk pr --all` against both resolves, so
this step is mainly for catching anything specific to your local
environment (e.g., integration tests with services the workflow
runner doesn't have):

```bash
bin/dk up      # start Docker services if needed
bin/dk pr      # full PR validation
```

### 5. Approve and Merge

The CI pipeline validates against the upgraded resolve. Once green,
approve and merge.

## Troubleshooting

### Floor-Resolve Validation Fails But Upgraded-Resolve Passes

Our code uses an API that doesn't exist at the floor of some
dependency. Identify the failing call (test or import error
points at it) and either:

- Bump the floor of that dep to a version where the API exists, or
- Use an older API that works at the floor

### Floor-Resolve Validation Hangs or OOMs

`uv sync --resolution lowest-direct --all-packages` can pull in
substantially older versions of heavy deps. If the floor resolve
exhausts memory or timeouts, the immediate fix is to bump floors
on the heaviest contributors (typically `transformers`, `torch`,
`pyarrow`) to known-working versions. The workflow's failure log
should point at where it stalled.

### `osv-scanner` Reports a CVE With No Fix Version (`--`)

The CVE is unpatched upstream. Options:

- If the dep is optional and unused in our path, document and defer
- If the CVE has a documented mitigation that doesn't require a
  version bump, apply it in our code
- If neither applies, monitor for a fix release; the weekly workflow
  will keep surfacing it

### Tests Fail After a Major-Version Bump

1. `bin/dk diagnose` to identify the failing tests
2. `bin/dk deps` to see which packages bumped majors
3. Read the upstream changelog for breaking changes
4. Either update our code to the new API or pin the dep below the
   bump with a comment explaining why and a tracking issue

### `uv sync` Fails With Resolution Errors

A dependency conflict was introduced upstream. Try:

```bash
uv lock --upgrade
uv sync --all-packages --all-groups
```

If resolution still fails, check `uv.lock` for conflicting version
constraints and resolve in `pyproject.toml`.

## The `dk deps` Command

`dk deps` is a thin wrapper around `bin/dep-diff.py` that compares
`[[package]]` blocks in `uv.lock` between two snapshots.

```bash
dk deps              # working tree vs main
dk deps <ref>        # working tree vs any git ref
dk deps --staged     # staged uv.lock vs HEAD
```

Output classifies changes into updated / added / removed and flags
major-version bumps.
